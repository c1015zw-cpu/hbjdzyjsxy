# 迎新网站制作工作流

这份流程用于后续制作或修改迎新攻略网站。每次发布到线上前，必须先得到明确确认。

## 1. 明确目标

- 确认学校名称、年份、受众、站点用途和是否为非官方攻略页。
- 确认需要展示的账号素材：头像、主页截图、昵称、公开数据和展示边界。
- 涉及报到时间、缴费、宿舍、军训、接站、电话等信息时，只使用学校正式通知或官方入口作为依据。

## 2. 收集与核验

- 优先核验学校官网、招生网、迎新系统、智慧校园和学院通知入口。
- 记录学校地址、咨询电话、官方入口链接和来源日期。
- 不把群聊、截图、经验帖里的未核验信息写成确定结论。

## 3. 内容拆分

- 把新生问题拆成可点击或可切换模块：到校前、报到日、宿舍、开课前、安全避坑、官方入口。
- 每个模块只保留可执行信息，减少空泛说明。
- 高风险事项加入核验提醒：缴费、贷款、兼职、电话卡、快递、证件、验证码。

## 4. 视觉设计

- 从用户提供素材和参考网站中提取主色、卡片形状、背景质感、导航形式和互动节奏。
- 第一屏必须直接出现学校名、站点用途和核心行动入口。
- 移动端优先：底部快捷导航、足够大的点击区域、清晰折行、首屏快速进入任务。

## 5. 页面实现

- 静态站点优先使用 `index.html`、`styles.css`、`script.js`，便于 GitHub Pages 部署。
- 常用组件顺序：顶部导航、首屏、报到流程、任务卡、清单、官方入口、账号介绍、工作流。
- 勾选清单使用 `localStorage`，只保存当前浏览器本地状态。

## 6. 验证

- 检查中文编码、图片路径、链接、按钮跳转、任务切换、清单保存。
- 检查手机端是否横向溢出，底部导航是否遮挡正文，按钮文字是否可读。
- 检索并清除项目混用痕迹、对话痕迹、错误学校名和无关素材引用。

## 7. GitHub 与部署规则

- 代码可以提交到 GitHub，但部署必须单独确认。
- GitHub Pages workflow 只允许 `workflow_dispatch` 手动触发，不允许 `push` 自动部署。
- 每次准备发布前，先说明将发布的提交、主要改动和目标网址，等待确认后再触发部署。
- 未获得确认时，只能完成本地修改、提交或推送代码，不能运行部署 workflow。
